The Challenge: Data Chaos in a Global Crisis

During the height of the COVID-19 pandemic, our healthcare client faced an unprecedented data challenge. They needed to ingest, process, and analyze rapidly changing data from dozens of disparate public health APIs and datasets. The data was inconsistent, schemas changed without notice, and the volume was immenseâ€”over 10 million new records daily. Their existing manual processes were completely overwhelmed, leading to days-long delays in analysis. The mission was clear: build a fully automated, scalable, and resilient data platform to provide timely, actionable insights for critical public health decisions.

Architectural Strategy: An Event-Driven, Serverless-First Pipeline

I designed a serverless-first, event-driven architecture to maximize scalability and minimize operational overhead. The core principle was to build a reactive system where data arrival itself would trigger the downstream processing steps. This approach allowed us to handle unpredictable bursts in data volume without manual intervention. We leveraged a combination of schedulers and event triggers to create a pipeline that was both robust for batch processing and agile enough for real-time updates, ensuring decision-makers always had the most current data at their fingertips.

Technical Implementation Details

1. Automated API Extraction Layer

To handle the numerous, unreliable APIs, I built a resilient extraction layer using AWS Lambda functions triggered by Amazon EventBridge (CloudWatch Events) schedulers. Each function was responsible for a specific API, encapsulating its unique logic, authentication, and error handling. Data was landed in its raw JSON or CSV format directly into an S3 "ingestion" bucket, providing a fault-tolerant and auditable entry point into our system.

2. Scalable Transformation with Glue and Spark

The core transformation logic was built using AWS Glue for its serverless Spark environment. I authored complex Python/PySpark jobs to read the raw, multi-format data from S3, apply schema-on-read, perform extensive cleaning (handling nulls, standardizing date formats), and join disparate datasets. By leveraging Spark's distributed processing, we could process the entire 10M+ daily record volume efficiently, achieving a 30% reduction in data preparation time compared to previous methods.

3. Unified Analytics with Databricks SQL & Hive Metastore

To enable rapid ad-hoc querying and analytics for data scientists, we configured Databricks SQL on top of our curated S3 data lake. We used AWS Glue Data Catalog as a Hive-compatible metastore, which provided a unified, SQL-addressable view over all our cleansed datasets. This allowed analysts to write standard SQL against petabytes of data without needing to manage the underlying infrastructure.

4. Orchestration and Real-time Visualization

The entire multi-step workflow was orchestrated using Managed Airflow (MWAA). For real-time updates, a parallel path was established using Kinesis to stream critical events directly to AWS QuickSight. I developed interactive dashboards in QuickSight, featuring geospatial maps, trend lines, and KPI trackers, which provided stakeholders with immediate, actionable insights into COVID-19 trends.
